css flex layout

flex 布局

css 2.1 定义了四种布局模式–块状布局，行内布局，table布局,position 布局.
这里将要介绍一个新的布局模式 –flex 布局。

flex 布局是为更复杂的应用和网页而设计的。

main size:在main dimension上flex item的宽或高
cross size:在cross dimension上flex item的宽或高

flex container

是根据内容建立在新的flex格式化上下文，用来替代块状布局。因此有些属性在flex布局中是不起作用的。
1.多列布局中的’column-*’属性
2.’float’和’clear’
3.’vertical-align’
4.’:first-line’和’:first-letter’伪元素

flex items

flex item 的display值已经是块状化的(blockified).

flex item margins and paddings

相邻的两个flex item 的margins不会折叠(collapse)

flex item应该完全避免在paddings或者margins上使用百分比,因为他们在不同浏览器中会有不同的表现.

主轴对齐中的margin auto

对每条flex line来说，如果主轴(main-axis)剩余的空白是正数,并且在在主轴上的至少有一条的margin值为auto,则将剩余的空白平分给margin.相当于给主轴上的每个items设置’justify-content’

交叉轴对齐中的margin auto

如果一个flex item交叉轴(cross-axis)margin值为auto:
1.如果他的交叉轴大小的边缘(outer cross size)小于他的flex-line的cross size,把他们大小之差平分给margins的值为auto(与主轴类似)
2.不然，如果(在交叉轴上)的block-start或者inline-start的margin是auto,让他等于0.设置对面margin的值,让他的交叉轴大小的边缘(outer cross size)等于他的flex-line的cross size.(*)

关于《webkit技术内幕》的读书笔记

前言

读这本书的目的是想了解有关浏览器的相关知识，但可能对于我而言，读的有些困难。大致浏览了下涉及的范围还挺广的.可能水平原因，收获不是很大，再过段时间，再读一次，且看如何。

webkit的渲染过程

从网页URL到构建完DOM树这个过程，数字表示基本顺序，这个过程可能重复并且交叉

1.当用户输入网页URL的时候，WebKit调用其资源加载器加载该URL对应的网页。
2.加载器依赖网络模块建立连接，发送请求并接收答复。
3.WebKit接收到各种网页或资源的数据，其中某些资源可能是同步或异步获取的。
4.网页被交给HTML解释器转变成一系列的词语(Token)
5.解释器根据词语构建节点(NODE),形成DOM树。
6.如果节点是javascript代码的话，调用javascript引擎解释并执行.
7.javascript代码可能会修改DOM树的结构。
8.如果节点依赖其他资源，例如图片、CSS、视频等，调用资源加载器来加载他们，但是它们是异步的，不会阻碍当前DOM树的继续创建；如果是javascript资源URL(没有标记异步方式),则需要停止当前DOM树创建，直到javascript的资源加载并被javascript引擎执行后才继续DOM树的创建。

WebKit利用CSS和DOM树构建RenderObject树直到绘图上下文

1.CSS文件被CSS解释器解释成内部表示结构。
2.CSS解释器工作完成之后，在DOM树附加解释后的样式信息，这就是RenderObject树。
3.RenderObject 节点在创建的同时，WebKit会根据网页的层次结构创建RenderLayer树，同时构建一个虚拟的绘图上下文。其实这中间还有很复杂的内部过程。

根据上下文来生成最终的图像，主要依赖2D和3D图形库

1.绘图上下文是一个与平台无关的抽象类，它将每个绘图操作桥接到不同的具体实现类，也就是绘图具体实现类。
2.绘图实现类也可能有简单的实现，也可能有复杂的实现，在Chromium中，他的实现相当复杂，需要Chromium的合成器来完成复杂的多进程和GPU加速机制。
3.绘图实现类将2D图形库或者3D图形库绘制的结果保存下来，交给浏览器来同浏览器界面一起显示.
